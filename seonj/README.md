<h1> 제 2장 객체지향 프로그래밍 </h1>

조건 vs 정책

- 조건이란 어떠한 사항의 부합 여부를 결정함
- 정책이란 조건에 부합할 때의 룰을 의미함

도메인 구조를 따르는 프로그램 구조

![image](https://user-images.githubusercontent.com/108404687/220862426-338a829e-054b-43cc-9c59-5e129c845024.png)

- 클래스의 이름은 대응하는 도메인의 개념 이름과 유사하거나 동일해야 함
- 클래스 사이의 관계도 개념 사이의 관계와 유사하게 만들어서 프로그램 구조를 쉽게 혜상할 수 있게 해야함

클래스 구현 혹은 개발된 클래스 사용시 중요한 것

1. 클래스의 경계를 내외부를 구분 지어야함

= 어떤 부분을 공개하고 어떤 부분은 감출 것인가?

1. 자율적인 객체
  1. 객체는 상태와 행동을 지녀야함; 접근 제어를 통해 외부 간섭 줄여야함
  2. 데이터와 기능을 객체 내부로 함께 묶어야 함 = 캡슐화
  3. 외부 내부 접근의 경계선 = 인터페이스와 구현의 분리법칙
2. 프로그래머의 자유
  1. 클래스 작성자(초기 작성자?)는 데이터 타입을 새로 추가
  2. 클라언트 작성자는 클래스 작성자가 추가한 데이터 타입 활용

상속과 다형성

- 컴파일 시간 의존성과 실행 시간 의존성

= 코드 의존성과 실행 시점의 의존성이 다를 수 있음

- 차이에 의한 프로그래밍

= 부모 클래스와 다른 부분만 추가해서 새로운 클래스를 만드는 방법

- 상속과 인터페이스

= 부모의 인터페이스는 모두 자식 클래스가 물려받음

- 인터페이스와 다형성
  - 다형성 : 메시지를 수신했을 때 객체의 타입에 따라 응답이 달라질 수 있는지
  - 다형성은 실행될 메세드를 실행 시점에 결정하기 때문에 '지연 바인딩' or '동적 바인딩' 이라 부름
  - 구현 공유가 아닌 순수한 인터페이스만 공유할때는 추상 클래스가 아닌 인터페이스 사용

추상화와 유연성

- 추상화를 사용하면 요구 사항의 정책을 높은 수준에서 서술하고 설계가 유연해짐
- 다만 추상 클래스와 인터페이스 간의 트레이드 오프가 존재한다

코드 재사용 및 상속

- 코드 재상용을 위해 상속보다는 합성을 사용
- 상속은 캡슐화를 위반

      - 부모 클래스의 내부구조를 알아야함
      - 부모 클래스 변경시 자식 클래스도 함께 변경될 가능성 높음
      - 설계의 유연함을 방해함
    - 상속 : 인터페이스에 정의된 메시지를 통해서만 코드 재사용
      - 인터페이스 재사용시 다형성을 위해 상속과 합성을 조합해서 사용
